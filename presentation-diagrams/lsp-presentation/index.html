📸 Snapshot LSP

with diagrams by Copilot

The problem we're solving

  • Old code relied on mutable data structures
  • Even with mutexes, we had data races
  • Code was becoming unmanageable
  • Hard to integrate another client
    (MCP or LSP plugin with API access)
The original TypeScript language server had fundamental issues with concurrent access to mutable data structures. Even though we used mutexes, the complexity was growing and we were still hitting race conditions.

Race condition example: ScriptInfo

This diagram shows a specific race condition that would occur with the old mutable ScriptInfo objects. The ScriptInfo stores file content and a LineMap for position conversions. When an LSP request comes in, we convert line/char to position, run the language service operation, then convert the result back. But if a file change happens concurrently, the LineMap changes mid-operation, causing incorrect position mappings.

The solution: Immutable snapshots

A snapshot is immutable state that can be read to serve a request or cloned to produce a new state capable of serving that request.

📦 Programs were already immutable - this gets us pretty far!

But we needed to make these things immutable too:

  • 📄 File state (both open and closed)
  • 🗂️ Projects (which ones are loaded and what do they contain)
  • ⚙️ ParsedCommandLine (loaded state)
The key insight was that TypeScript Programs were already immutable, which gave us a good foundation. But we needed to extend this immutability to the surrounding infrastructure - file states, project configurations, and command line parsing results.

Operating on snapshots: No side effects

When operating on a snapshot, we must produce no side effects:

❌ Forbidden

  • 📁 File watching
  • 📝 Logging (ideally)
  • 🚀 Project loading
  • 📦 ATA (Automatic Type Acquisition)

✅ Pure operations

  • 🧮 Use language service
  • 📚 Populate internal caches

Result: Concurrent requests can operate safely on the same snapshot! 🎉

This is crucial - when we're operating on a snapshot to serve an LSP request, we cannot perform any operations that would modify global state. No file system operations, no logging that could block, no loading new projects. The snapshot contains everything we need, and operations on it must be pure functions.

LSP server hierarchy

This diagram shows the complete hierarchy of the LSP server architecture. At the top is the Session which manages ref-counted caches, pending changes, file system access, and open file state. The Current Snapshot contains immutable copies of the cached file system, open file state, project collection, and config file registry. Each project has its own file watcher registrations and contains the TypeScript Program, Language Service, and CheckerPool. The ConfigFileRegistry manages parsed command line configurations and their associated file watchers.

Cloning a snapshot

This diagram shows the process of creating a new snapshot from pending changes. First, the Session applies its pending changes to the overlay state to create a file change summary. This summary is then combined with the current snapshot, ref-counted caches, and a snapshot of the open file state through the cloning process to produce a new immutable snapshot. This process ensures that all changes are properly captured while maintaining immutability.

The dirty.Map pattern

How do we efficiently track changes to immutable data?

📝 dirty.Map - A pattern for layering changes on top of immutable base data

  • 🏗️ Base Map: Immutable foundation data
  • 📋 Dirty Entries: Pending changes/overrides
  • 🔄 Read-through: Check dirty first, then base
  • ✨ Finalization: Creates new immutable map with changes applied
The dirty.Map pattern is fundamental to our snapshot system. It allows us to efficiently track changes without modifying the original immutable data. When we need to make a change, we clone the entry to the dirty layer. Reads check the dirty layer first, falling back to the base map. When we finalize, we create a completely new immutable map with all changes applied.

dirty.Map - Clean state

This shows a dirty.Map in its clean state - no pending changes. All data comes from the immutable base map on the left. The dirty entries column on the right is empty because no modifications have been made yet. Read operations access the base map directly since there are no overrides to consider.

dirty.Map - With pending changes

Here we see what happens when a change is made. The original entry for key2 is cloned to the dirty entries column with the new value. The original entry in the base map becomes dimmed since it's now overridden. Read operations for key2 now come from the dirty entry instead of the base map, while unmodified entries like key1, key3, and key4 still read from the base map directly.

dirty.Map - Finalization process

The finalization process shows how we create a new immutable map from the dirty state. The top half shows the current state with base map and dirty entries. During finalization, we create a completely new immutable map where dirty entries override their corresponding base entries, and unchanged entries are carried forward. The result is a new immutable foundation that can serve as the base for future changes.

dirty.MapEntry Change()


entry.Change(func(file *diskFile) {
    file.content = content
    file.hash = sha256.Sum256([]byte(content))
    file.needsReload = false
})
                

func (e *MapEntry[K, V]) Change(apply func(V)) {
	if e.delete {
		panic("tried to change a deleted entry")
	}
	if !e.dirty {
		e.value = e.value.Clone()
		e.dirty = true
		e.m.dirty[e.key] = e
	}
	apply(e.value)
}

Builder Pattern: Snapshot Construction

When we clone a snapshot, we create three specialized builders that use dirty maps to track changes. Each builder manages a different aspect: snapshotFSBuilder for filesystem changes, projectCollectionBuilder for project updates, and configFileRegistryBuilder for config management with dual dirty maps. The three-phase lifecycle shows initialization, applying changes, and finalization. The key insight is that dirty maps only store what changed - the original immutable data is referenced, not copied. This enables efficient copy-on-write semantics across the entire snapshot system.

LSP request processing flow

This diagram shows the complete flow when an LSP request comes in. The process starts with GetLanguageService, which calls flushChanges to check for pending file or ATA changes. If changes exist, UpdateSnapshot is called with mutex locking. The snapshot is cloned, changes are applied, and ref counts are updated. If no changes exist, the current snapshot is used with a read lock. The system then checks if the project is loaded and up-to-date - if not, another snapshot update may be triggered. Finally, background tasks handle logging, file watching, and ATA updates.
